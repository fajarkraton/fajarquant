**Blocker discovered:** `preprocess_weights_tl2` shape gate:
```python
if K == KEMD:        # 1536
    BM3, BY3 = BMEMD, BYEMD
elif K == KGQA:      # 4096
    BM3, BY3 = BMGQA, BYGQA
else:
    raise NotImplementedError
```

Our test shape K=256 hits `NotImplementedError`. The encoder is
**model-specific** — supports only the two K dimensions used by upstream's
target models (typical Llama variants). Fajarquant_mini's (256, 256) shape
was generated by adapting `codegen_tl2.py` for custom dimensions; the
**encoder side was never adapted upstream-side** — it's a gap in microsoft/
BitNet's pipeline that we can't simply port from.

### Step 2: Sub-experiment — run upstream at K=1536 to extract structure (~30min)

Set M=256, K=1536 (BMEMD, KEMD). Encoder runs. Debug output:
```
sign_inner_by_weight shape=(32, 8)
func_weight pieces: 8, each shape=(32, 1)
i=0: min_top shape=(16, 1), min_bot shape=(16, 1)
combine after i=0: shape=(16, 1)
```

Confirms: `sign_inner_by_weight` slices to (32 rows, 8 sign-cols). Each
spatial column `i ∈ [0..7]` maps to bit positions `15-2i` (top half rows
0..15) / `14-2i` (bot half rows 16..31) in the output u16. **Matches our
derivation exactly** (Tier 2E findings §1.3).

This was already known. The sub-experiment confirmed it but didn't reveal
the specific bug in our encoder.

## 2. New finding — kernel drops 2 triplets per kouter

While reading kernel inner loop structure during Step 1:

```c
// codegen-emitted three_tbl_impl_256_256:
for (int k = 0; k < KK / 8; k++) {       // KK/8 iterations
    __m256i vec_sign = vec_signs[k];
    // 4 ai-blocks per inner-k iteration
    // ...
}
```

For our config: `kk = bbk // 3 = 42`. Inner loop `for (k = 0; k < 42/8 = 5; ...)`.

**Per kouter: 5 inner-k iterations × 4 ai × 2 nibbles = 40 triplets covered.
2 triplets (idx 40, 41 within kouter) are SILENTLY DROPPED.** Plus the
`k mod 3` tail (1 weight at k=126,127 for kouter 0; k=254,255 for kouter 1)
is also dropped. Total dropped per row: 2 kouters × (2 triplets × 3 + 1 tail
= 7 weights) = **14 weights per row dropped**.

Plus: K=256 has K mod 3 = 1, so even the scalar drops 1 weight per row.
Net difference between scalar processing and kernel processing: 13 weights
per row dropped only by kernel.

This is INFRASTRUCTURAL to the kernel — not fixable by encoder changes.
Either:
1. Accept the silent drop (mark kernel as approximate within ε for these
   shapes), or
2. Use a kernel built for K-shape divisible-by-(BBK*8) so KK/8 floor doesn't
   drop triplets, or
3. Pad K to a friendlier value at inference time.

## 3. Diff attribution analysis

Measured for row 0 (j%3=0 class): `tl2 - scalar = +32`.

Predicted contribution of dropped weights for row 0:
- Weight pattern: `(0+i)%3` cycles (0, 1, -1, 0, 1, -1, ...)
- Activation pattern: `(i%17) - 8` cycles (-8, -7, ..., 8, -8, ..., 8)
- Dropped k for kouter 0: k = 120..125 (triplets 40, 41 of kouter 0)
- Dropped k for kouter 1: k = 248..253

```
Sum over dropped k of w[0,k] * act[k]:
  k=120..125: 0*(-7) + 1*(-6) + (-1)*(-5) + 0*(-4) + 1*(-3) + (-1)*(-2) = -2
  k=248..253: (-1)*2 + 0*3 + 1*4 + (-1)*5 + 0*6 + 1*7              = +4
  total dropped contribution                                          = +2
```

**If dropped-triplets were the only bug: predicted diff = +2.
Measured diff = +32. Discrepancy = +30 unaccounted for.**

So dropped-triplets explain ~6% of observed diff. **The remaining ~94% is
from a separate layout bug** (likely in row_hi_lo or column-ordering, both
flagged as remaining hypotheses in Tier 2E findings §3).

### Option B — Re-attempt F.11 with printf-instrumented kernel

Build kernel with `-DDEBUG_TRACE_SIGNS=1` macro that dumps every sign byte
read with its (i, as, lane, byte_offset) coordinate. Run 1 known-input test,
diff predicted vs actual reads. Time estimate: ~3-4h. Higher confidence of
closure than Branch X-real because it gives empirical ground truth from the
RUNNING kernel rather than reasoning about layout.

Cost: 3-4h.
Outcome: ~70% chance of full F.11 closure.

### Option C — Replace TL2 with a simpler custom AVX2 kernel

Skip the TL2-format complexity entirely. Write a hand-coded AVX2 BitLinear
kernel that doesn't use the TL2 LUT trick (just direct ternary-times-int8
multiplication). Likely 2-3× speedup over scalar Rust (vs TL2's 5-7×) but
fully under our control.

Cost: ~1-2 GPU-day equivalents in CPU dev work. Substantial.
Outcome: Removes the upstream-dependency and hypothesis-stack entirely.

**Recommendation: Option A (stop F.11)** unless paper v2 reviewer specifically
requests bit-exact TL2 parity. F.13.3 verdict already commits FajarOS to
scalar-default; F.11 closure was never load-bearing for any deployment.
